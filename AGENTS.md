# AGENTS.md — TPC-H GPU baseline

Benchmark harness comparing GPU/CPU query engines on **TPC-H** (single node,
2× RTX 5090 / 32 GB, 256 cores, 1 TB RAM). Start with `README.md` for the full
story; this file is the quick agent orientation, headline results, and
results-data contract.

## Headline results (single 32 GB RTX 5090)

Total completed-query seconds by engine × scale factor, `(#OK / 22)`
(raw per-query rows: `results/all_results.csv`):

| SF | Polars-CPU | Sirius | Polars-GPU | Spark-RAPIDS |
|---:|-----------:|-------:|-----------:|-------------:|
| 30 | **25** (22) | 43 (22) | 34 (22) | 333 (22) |
| 50 | **39** (22) | 52 (22) | 46 (22) | 368 (22) |
| 100 | 96 (22) | **91** (22) | 104 (21) | 504 (22) |
| 300 | **243** (22) | 344 (21) | 774 (21) | 870 (22) |
| 500 | **366** (22) | 439 (17) | 2645 (22) | 1102 (22) |

- **Polars-CPU wins at every scale** — 122 threads + 1 TB RAM, no PCIe spill.
- **Sirius** — fastest GPU engine where it fits (on the 17 queries it completes at
  SF500: Sirius **439 s** vs RAPIDS **640 s** vs Polars-CPU **164 s**), but the
  heaviest hash-joins **OOM the 32 GB GPU**: q8 at SF300; q3/q8/q9/q10/q21 at SF500.
  A data-size wall (they run fine at smaller SF), not a bug. Multi-GPU
  (`num_gpus: 2`) is the way to push it out.
- **Polars-GPU** (cudf-polars) — tracks CPU at small SF, **spill-bound** at large
  SF (2645 s @ SF500); q18 config-fails (`target_partition_size`) at SF100+.
- **Spark-RAPIDS** — high fixed per-query Spark overhead (totals flat, then grow),
  but the most **robust** GPU engine: all 22 at every SF, spilling gracefully to
  its 200 GB host store (needs ramdisk shuffle scratch at large SF — `SPARK_SCRATCH`).
- Result **row counts match across engines at each SF** — a mismatch = correctness bug.

## Engines & runners (one process per query + disk watchdog)

| engine key | dir | runner |
|------------|-----|--------|
| `sirius` (Sirius GPU, DuckDB ext.) | `sirius/` | `run_sirius.sh` |
| `rapids` (Spark + RAPIDS GPU) | `rapids/` | `run_tpch_safe.sh` |
| `polars_cpu` (Polars streaming) | `polars/` | `run_polars.sh` |
| `polars_gpu` (cudf-polars) | `polars/` | `run_polars_gpu.sh` |

Data is TPC-H parquet on the **ramdisk** (`/dev/shm`), generated by
`rapids/nds_h_pipeline.sh <SF> <PARALLEL> <BATCH> <out_dir>`. Runners read the
same `results/queries/stream_qualification.sql` and take env overrides
`TPCH_PARQUET`, `TPCH_SF` (scale-factor label, default 500), `SPARK_SCRATCH`, `ENGINES`.

**No per-run CSVs.** Each runner writes a throwaway temp CSV, then folds its 22
rows into `results/all_results.csv` via `merge_results.py <engine> <sf> <run_csv>`
(an idempotent upsert of the `(engine, scale_factor)` slice) and deletes the temp.
Set `TPCH_MERGE=0` to keep the temp instead of merging.

## Scale sweeps (query time vs data size)

- `sirius/run_scale_sweep.sh [SFs]` — Sirius at SF30/50/100/300.
- `run_scale_sweep_all.sh [SFs]` — rapids + polars_cpu + polars_gpu (gen each SF
  once, run all with `TPCH_SF` set, free it). `ENGINES=…` selects engines;
  `SPARK_SCRATCH=/dev/shm/...` gives Spark ramdisk shuffle scratch (RAPIDS, large SF).

The runners self-merge, so the sweeps produce **only** `results/all_results.csv` —
no per-SF intermediates.

## Results data contract — `results/all_results.csv`

**The single source of truth.** One row per `(engine, scale_factor, query)`.
Summary / matrices / per-engine scaling are all pivots of this — do NOT add
parallel summary CSVs; pivot this instead.

| column | type | values / meaning |
|--------|------|------------------|
| `engine` | string | `polars_cpu` · `polars_gpu` · `rapids` · `sirius` |
| `scale_factor` | int | `30` · `50` · `100` · `300` · `500` (TPC-H SF ≈ GB raw) |
| `query` | string | `query1` … `query22` |
| `status` | string | `OK` · `FAIL` (engine error; **GPU OOM** shows here with an "OOM retry limit" message in `rows_or_error`) · `KILLED_DISK` (disk-watchdog kill) · `TIMEOUT` (per-query timeout) |
| `seconds` | float | per-query wall-clock, **cold**, engine startup excluded; time-to-failure on error; `NA` on a watchdog kill |
| `rows_or_error` | int / string | result **row count** when `OK`, else a short error string |

```csv
engine,scale_factor,query,status,seconds,rows_or_error
polars_cpu,100,query1,OK,2.308,4
sirius,500,query9,FAIL,259.488,INTERNAL Error: ... GPU pipeline task exceeded maximum OOM retry limit (100) for
```

Pivots:
```bash
duckdb -c "SELECT engine,scale_factor,count(*) FILTER(status='OK') ok,
  round(sum(seconds) FILTER(status='OK'),1) total_s
  FROM 'results/all_results.csv' GROUP BY 1,2 ORDER BY 1,2"          -- summary
duckdb -c "PIVOT 'results/all_results.csv'
  ON engine||'_sf'||scale_factor USING first(seconds) GROUP BY query"   -- matrix
```

## Notes for agents

- Row counts (`rows_or_error` when `OK`) vary with scale factor but must **match
  across engines at the same SF** — a mismatch means a correctness bug.
- Not committed (regenerate): the `sirius/sirius/` clone + `.pixi/` env, python
  venvs, the RAPIDS jar/conda env, and the SF parquet (all gitignored).
- Container quirks: `io_uring` is blocked (Sirius uses kvikio, `KVIKIO_COMPAT_MODE=ON`);
  drop empty 0-row parquet part-files before a Sirius run (its GPU reader errors on them).
